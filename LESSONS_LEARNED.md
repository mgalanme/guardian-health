# Lessons Learned — GUARDIAN-Health

**Agentic AI Bootcamp — Case Study 3**
**Author:** Martín Galán
**Date:** April 2026

---

These are the lessons learned building GUARDIAN-Health from scratch: from the first line of the audit trail schema to the Solace cross-framework governance consumer. They are ordered by the phase in which they emerged, not by importance. All of them are important.

---

## 1. Architecture and Governance

**The audit trail is not a feature. It is the backbone.**

The most consequential design decision in the project was placing the audit trail as a first-class field in the LangGraph `GuardianState` TypedDict, not as a separate logging system. When the audit trail is external to the agent state, it captures what the agent did. When it is inside the state, it captures what the agent was thinking at each step. The difference matters enormously for regulatory compliance and forensic investigation. If you are building an agentic system in a regulated domain, design your state first and put the audit trail inside it before you write a single agent.

**Retroactive governance is not achievable. This is not an opinion.**

You cannot add governance to an agentic system after it is built. The specific things you cannot recreate are: the reasoning trace of a past LLM call (you can only record it at the time it happens), the provenance of a data item that fed an agent context (you must capture it at ingestion), and the chain of custody for a decision (you must record the human who approved it when they approve it, not later). Every hour spent building without governance infrastructure is an hour of unrecoverable debt.

**Governance lives in the bus topology, not in the agent code.**

This was the most architecturally interesting insight of the Solace implementation. When each module publishes to `guardian/v1/audit/{module}/{event_type}`, a single consumer can capture all governance events regardless of whether they were generated by LangGraph, CrewAI, or a direct API call. The governance policy is the topic hierarchy. Change the hierarchy, and you change the policy — without touching any agent code. This is the correct level of abstraction for cross-framework governance.

**TOGAF and DMBOK are not overhead. They are the scaffolding.**

Starting with TOGAF Preliminary Phase (governance principles before any code) and DAMA DMBOK2 (data domain classification before any schema) felt like it slowed things down at first. It did not. Every ambiguous design decision during implementation was resolved by referring back to a principle defined in the architecture phase. The principles were: no component without an owner, no decision without traceability, pseudonymisation mandatory before any LLM call, and human oversight proportional to risk. These four principles resolved dozens of micro-decisions during implementation without requiring a new design session each time.

## 2. Data and Pseudonymisation

**GDPR Article 9 is a hard design constraint, not a compliance checkbox.**

The first time we tried to pass clinical data to Groq without pseudonymisation, the governance layer immediately rejected it. That rejection was the system working correctly. In a real hospital environment, sending identifiable patient data to an external LLM would be a serious GDPR violation with real consequences. The two-layer pseudonymisation scheme (permanent UUID per patient, session-scoped alias per agentic session) is more complex than simple anonymisation, but it preserves the clinical utility of the data while keeping identifiers out of the LLM context. The complexity is justified.

**Data quality in agentic systems has a dimension that DMBOK2 does not fully cover: contextual accuracy.**

Standard data quality dimensions (completeness, consistency, timeliness, accuracy) apply to agentic systems, but there is an additional dimension specific to agents: is the data in the agent's context representative of the actual situation? A patient's creatinine reading from three days ago is accurate data, but it may not be contextually accurate for an agent making a real-time decision. We addressed this by including `data_as_of` timestamps in every signal and degrading confidence scores for stale data. This dimension deserves a formal name and a place in data quality frameworks for AI systems.

**Graph databases are not optional for clinical pharmacovigilance.**

The detection of `digoxin + low potassium → arrhythmia risk` is trivially expressible in Cypher:
```cypher
MATCH (p:Patient)-[:PRESCRIBED]->(m:Medicine)
WHERE toLower(m.display) CONTAINS 'digoxin'
MATCH (p)-[:HAS_OBSERVATION]->(o:Observation)
WHERE toLower(o.display) CONTAINS 'potassium'
  AND o.interpretation IN ['L', 'LL']
```
The equivalent in a relational database requires a multi-table JOIN that is both harder to write and harder to explain to a clinician. In pharmacovigilance, the signal patterns are inherently graph-shaped: drug A + condition B + lab value C → risk D. Use a graph database.

**Duplicate SNOMED codes in synthetic data caused silent failures in Neo4j MERGE operations.**

When two medicines share the same SNOMED code (as happened in our initial synthetic data generator), Neo4j's `MERGE` collapses them into a single node, and the last `SET` wins. This caused ibuprofen to be effectively invisible in the graph, which caused the NSAID signal detection to fail silently. The lesson: synthetic data generators must enforce domain integrity constraints (unique SNOMED codes, unique ICD-10 codes) the same way production systems would. Silent failures in synthetic data are harder to debug than silent failures in real data because you cannot check the source.

## 3. Agentic AI Patterns

**The right framework for each module is determined by the governance requirements, not by familiarity.**

We used LangGraph for VIGIL (stateful flow with HITL interrupts), CrewAI for ASSESS (multi-agent crew with typed outputs), and AutoGen for adversarial debate (formally structured agent conversations). The selection criterion in each case was: which framework makes the governance requirements easiest to implement? LangGraph won for VIGIL because its checkpointing and interrupt model are exactly what HITL requires. CrewAI won for ASSESS because its task output typing with Pydantic guarantees that every evaluation step produces a structured, auditable artefact. Choosing frameworks based on familiarity rather than fit is a common mistake in agentic projects.

**LLM output parsing in multi-agent crews requires defensive normalization.**

The synthesis agent in the CrewAI crew sometimes returned `"REPORTABLE"` instead of `True` for the `reportable` field, and a placeholder UUID instead of a real one, and raw JSON instead of a narrative. None of these are bugs in CrewAI; they are natural variation in LLM outputs. The parser must normalise for all of them. Expecting clean, schema-conformant JSON from an LLM in a multi-turn crew without defensive normalisation is optimistic. The practical rule: treat every LLM output as potentially malformed, and write normalisation logic that handles the three most common failure modes (wrong type, placeholder value, absent field) before shipping.

**The `handler_done` pattern for async consumers is only correct for single-message flows.**

When VIGIL can emit multiple signals (and it can — a patient might have both a critical drug interaction and an anomalous lab trend), the current `handler_done.set()` in the `finally` block terminates the consumer after the first message. The fix is to include `signal_count` and `signal_index` in the published payload so the consumer knows how many messages to expect. This is a known pattern in event-driven systems (the "end of batch" marker), but it is easy to miss when designing for the single-signal case first.

**Groq's free tier rate limits matter for multi-agent crews.**

The ASSESS crew makes four sequential LLM calls (one per agent). With Groq's free tier limits (~12k tokens/minute for llama-3.3-70b), a crew evaluation that uses ~3k tokens total completes in 5-7 seconds. A pipeline with 12 patients evaluated simultaneously would hit rate limits. In production, either use a paid tier, implement exponential backoff with `tenacity`, or batch the evaluations with a delay between them. We used `tenacity` for retries but did not implement batching, which is the next engineering step.

## 4. Infrastructure and Operations

**Solace PubSub+ requires `nofile` ulimit of 1,048,576 to start correctly.**

The standard Linux default of 524,288 causes Solace to fail at startup with a warning about required system resources, followed by a hard shutdown. The fix is a single `ulimits` block in the Docker Compose service definition. This is documented in Solace's official requirements but easy to miss. Always check container resource requirements before assuming a startup failure is a configuration error.

**The `ready_event` pattern is the correct way to coordinate producers and consumers in event-driven systems.**

A fixed `time.sleep()` before publishing is fragile: it assumes the consumer will be subscribed within that window, which depends on host load, network latency, and broker state. The correct pattern is: each consumer signals a `ready_event` immediately after calling `receive_async()`, and the producer waits for all `ready_event`s before publishing. This is deterministic regardless of timing. The fixed sleep is an anti-pattern that occasionally works in development and fails in production under load.

**CrewAI modules cannot be imported in secondary threads without pre-importing in the main thread.**

CrewAI performs thread-unsafe initialisation during its first import (specifically in `EmbeddingConfigurator` and related utilities). If `from crewai import Agent` is called for the first time inside a `threading.Thread`, the import fails with an obscure exception that is not obviously related to threading. The fix is to pre-import all heavy modules (`crewai`, `langgraph`, `langchain`) in the main thread before spawning any secondary threads. Once a module is in `sys.modules`, importing it in a secondary thread is a safe, near-instant operation.

**Docker Hub rate limits can block development entirely if you are not authenticated.**

Several hours of work were spent debugging what appeared to be network connectivity issues, which turned out to be Docker Hub rate limiting unauthenticated pulls. The practical fix: always `docker login` before attempting to pull images, and maintain a local cache of the images your development stack requires. Images that are already local are never subject to rate limits.

**Separate test databases from production databases for governance tests.**

The `test_chain_integrity_detects_tampering` test intentionally corrupts an audit record to verify that the detection mechanism works. When this test runs against the production `audit_trail` table, the corruption persists and breaks the cryptographic chain for all subsequent verification runs. The lesson: governance tests that mutate immutable data structures must use an isolated schema or be wrapped in transactions that always roll back. This is ADR-007 in the project's decision record.

## 5. Process and Portfolio

**Architectural rationale before code is not a luxury. It is the only way to build coherent agentic systems.**

Each module in this project was preceded by a written explanation of why a particular pattern or technology was chosen, what alternatives were considered, and what the governance implications of the choice are. This slowed the initial implementation and produced a system where every component fits together and every decision can be defended to a regulator, an auditor, or a technical interviewer. The alternative — write code first, rationalise later — produces systems that work but cannot be explained.

**The three-role approach (Enterprise Architect, Data Architect, AI Architect) is not a framework. It is a discipline.**

In every design session for this project, three questions were asked simultaneously: what does this mean for the business capability model? what does this mean for the data domain and its governance? what does this mean for the agent's state and decision traceability? Systems built by teams where these roles are siloed tend to have excellent agent logic and terrible data governance, or excellent data governance and no explainability. Holding all three perspectives simultaneously, even when working alone, produces better architectures.

**Synthetic data that does not enforce domain integrity is not synthetic data. It is noise.**

The debugging sessions caused by duplicate SNOMED codes in the synthetic data generator cost more time than the original implementation of the generator. Domain integrity constraints (unique identifiers, valid code system values, correct relationship cardinalities) are not optional in synthetic data. They are the whole point of using synthetic data in the first place.

**Document decisions as ADRs, not as comments.**

An Architecture Decision Record in `docs/adrs/` is findable, searchable, and has a clear status (Accepted, Superseded, Deprecated). A comment in a Python file explaining why you chose PostgreSQL over Cassandra is invisible to anyone who does not happen to read that specific file. The seven ADRs in this project represent decisions that future maintainers or clients would ask about. Each one saves at least one meeting.
